## 📌 제4장 테스트 분석과 설계 (11문항) — 최다 출제

### 4.1 테스트 기법 분류 ★★★
| 분류 | 테스트 베이시스 | 핵심 |
|------|--------------|------|
| **블랙박스** (명세 기반) | 요구사항, 명세서 | 내부 구조 모름. 구현 바뀌어도 TC 유효 |
| **화이트박스** (구조 기반) | 소스코드, 아키텍처 | 설계/구현 후에만 TC 작성 가능 |
| **경험 기반** | 테스터의 지식·경험 | 명세 없이도 가능. 테스터 역량에 의존 |

**시험 포인트:** 요구사항에서 비즈니스 규칙이 명시되면 → **블랙박스**가 가장 적합 (경험 기반도 가능하지만 블랙박스가 "더 적합")

### 4.2 블랙박스 테스트 기법 ★★★★★ (K3 — 직접 풀어야 함)

#### 4.2.1 동등 분할 (EP)
- 같은 분할 안의 값은 **동일하게 처리**된다고 가정
- 각 분할에서 **1개만** 테스트하면 충분
- **유효 분할 + 비유효 분할** 모두 식별
- 분할은 **겹치면 안 되고, 빈 집합이면 안 됨**
- 커버리지 = 실행한 분할 수 / 전체 분할 수 × 100%
- **이치 초이스 커버리지(Each Choice):** 모든 분할 집합의 각 분할을 최소 1번 실행

**풀이 팁:**
1. 변수별 분할을 먼저 식별
2. 기존 TC가 어떤 분할을 커버하는지 확인
3. 커버 안 된 분할에 해당하는 값을 선택

#### 4.2.2 경계값 분석 (BVA)
- **정렬된 분할**에만 적용 가능
- 개발자가 경계에서 실수할 가능성이 높음

| 유형 | 커버리지 항목 | 예시 (분할: 0~5, 6~12, 13~) |
|------|------------|--------------------------|
| **2개 선택 (2-value)** | 경계값 + 인접 분할의 가장 가까운 값 | 5, 6, 12, 13 |
| **3개 선택 (3-value)** | 경계값 + 양쪽 이웃값 | 4, 5, 6, 11, 12, 13, 14 |

**3-value가 2-value보다 엄격** → "if (x≤10)"이 "if (x=10)"으로 잘못 구현된 경우 2-value로는 못 찾지만 3-value의 x=9로 찾을 수 있음

**풀이 팁:**
1. 분할 경계를 먼저 모두 나열
2. 2-value면 경계값만, 3-value면 양옆까지
3. 기존 TC가 커버하는 경계값을 체크
4. 커버리지 = 커버된 경계값 수 / 전체 경계값 수

#### 4.2.3 결정 테이블 테스팅
- **조건 조합 → 동작(결과)** 매핑
- 열(column) = 규칙(Rule), 행 = 조건/동작
- 표기: T(참), F(거짓), -(무관), N/A(실행 불가), X(동작 발생), 공백(동작 없음)
- 커버리지 항목 = **실현 가능한 열(규칙)**
- 커버리지 = 실행된 열 수 / 실현 가능한 열 총수 × 100%

**풀이 팁:**
1. 각 TC의 조건값이 어떤 규칙(열)에 매핑되는지 확인
2. 여러 TC가 같은 규칙을 커버하면 중복 → 커버리지 증가 안 함
3. **모순 식별:** 같은 조건 조합이 서로 다른 동작을 가진 두 규칙에 매핑되면 모순

#### 4.2.4 상태 전이 테스팅
- **상태 다이어그램:** 상태 + 전이(이벤트 [가드] / 동작)
- **상태 테이블:** 행=상태, 열=이벤트, 빈 셀=비유효 전이

| 커버리지 기준 | 항목 | 관계 |
|------------|------|------|
| 모든 상태 커버리지 | 각 상태 | 가장 약함 |
| **유효 전이 커버리지** (0-스위치) | 각 유효 전이 | **가장 많이 사용** |
| 모든 전이 커버리지 | 유효 + 비유효 전이 | 가장 강함 |

- 100% 유효 전이 → 100% 상태 보장
- 100% 모든 전이 → 유효 전이 + 상태 모두 보장

**풀이 팁:**
1. 다이어그램에서 모든 유효 전이를 나열
2. TC의 이벤트 순서를 따라가며 어떤 전이를 커버하는지 체크
3. **같은 이벤트라도 시작 상태가 다르면 다른 전이!**
4. 최소 TC 수 = 동일 TC에 동시에 나타날 수 없는 전이 그룹 수

### 4.3 화이트박스 테스트 기법 ★★★

#### 4.3.1 구문 커버리지
- 커버리지 항목 = **실행 가능한 구문**
- 커버리지 = 실행된 구문 수 / 실행 가능 구문 총수 × 100%
- 100% 구문 커버리지 = 모든 실행 가능 구문이 최소 1번 실행됨
- **100% 구문 ≠ 100% 분기** (ELSE 없는 IF문 등)
- 결함 있는 구문을 실행해도 **반드시 장애가 발생하는 것은 아님** (데이터 종속적 결함)

#### 4.3.2 분기 커버리지
- 커버리지 항목 = **분기** (제어 흐름 그래프의 간선)
- 무조건 분기 + 조건 분기 모두 포함
- 커버리지 = 실행된 분기 수 / 전체 분기 수 × 100%
- **100% 분기 → 100% 구문 보장** (역은 성립 안 함)

#### 4.3.3 화이트박스 테스팅의 가치
- **강점:** 명세가 모호/불완전해도 코드 구현 전체를 고려하므로 결함 탐지 가능
- **약점:** 요구사항 누락(구현되지 않은 기능)은 식별 못 함
- 정적 테스팅(코드 드라이 런)에도 사용 가능
- 블랙박스만으로는 코드 커버리지 측정 불가 → 화이트박스 커버리지가 **객관적 측정값** 제공

### 4.4 경험 기반 테스트 기법 ★★

#### 4.4.1 오류 추정
- 테스터의 지식으로 **오류, 결함, 장애**를 예측
- 입력/출력/논리/계산/인터페이스/데이터 관련 결함 예측
- **결함 공격:** 오류/결함/장애 목록을 만들어 체계적으로 테스트
- 오류 추정으로 예측할 수 있는 것: 개발자 실수, 코딩 결함, 시스템 장애
- **예측할 수 없는 것:** 결함의 근본 원인(예: 개발자가 세미나를 놓친 것)

#### 4.4.2 탐색적 테스팅
- 테스트 **설계 + 실행 + 평가를 동시에** 수행
- 세션 기반: 시간 제한 + **테스트 차터**(목적 포함)
- **유용한 상황:** 명세 부족, 시간 압박, 도메인 지식 풍부할 때
- 탐색적 테스팅 **중에** 다른 기법(EP 등) 사용 가능
- TC를 **사전에 작성하지 않음** (세션 중 생성)

#### 4.4.3 체크리스트 기반 테스팅
- 체크리스트로 테스트 컨디션 확인
- 항목: **개별적이고 직접 확인 가능**해야 함
- 포함하면 안 되는 것: 자동 점검 가능 항목, 시작/완료 조건, 너무 일반적인 항목
- 상위 수준 체크리스트 → 테스트 약간씩 달라짐 → **커버리지↑, 재현성↓**
- 정기적으로 **결함 분석 기반 업데이트** 필요

### 4.5 협업 기반 테스트 접근법 ★★

#### 4.5.1 사용자 스토리 작성
- **3C:** Card(매체), Conversation(대화), Confirmation(인수 조건)
- 형식: "[역할]로서 [목표]를 달성해 [비즈니스 가치]를 원한다"
- **INVEST:** Independent, Negotiable, Valuable, Estimable, Small, Testable
- **대화(Conversation)**가 공통된 이해 달성의 핵심

#### 4.5.2 인수 조건
- 이해관계자가 승인하기 위해 충족해야 하는 조건
- **시나리오 기반:** Given/When/Then 형식
- **규칙 기반:** 검증 목록 또는 입출력 매핑 표

#### 4.5.3 인수 테스트 주도 개발 (ATDD)
- 사용자 스토리 구현 **전에** TC 작성
- 고객 + 개발자 + 테스터가 함께 작성
- **명세 워크숍** → TC 작성 → 구현
- TC는 사용자 스토리의 **모든 특성을 다뤄야** 하고 **스토리를 벗어나면 안 됨**
- 순서: 긍정/유효 TC → 부정/비유효 TC → 비기능 TC

## 📌 제5장 테스트 활동 관리 (9문항)

### 5.1 테스트 계획 ★★★

#### 5.1.1 테스트 계획서
- 목적: 테스트 목적 달성 방법/일정 문서화, 기준 충족, 의사소통 수단
- 내용: 정황, 가정/제약, 이해관계자, 의사소통, 리스크 목록, **테스트 접근법**, 예산/일정
- 테스트 접근법에 포함되는 것: 테스트 레벨, 유형, 기법, 시작/완료 조건, 커버리지 수준 등
- **테스트 계획에 포함되지 않는 것:** 개별 TC의 상세 테스트 데이터

#### 5.1.2 반복주기/릴리스 계획에 대한 테스터 기여
- 릴리스 계획: 테스트 가능한 사용자 스토리/인수 조건 작성, 리스크 분석 참여, 노력 추정, 접근법 결정
- 반복주기 계획: 구체적 리스크 분석, 테스트 용이성 판단, 업무 분류, 기능/비기능 측면 식별

#### 5.1.3 시작 조건과 완료 조건 ★★★
| 시작 조건 (Entry Criteria) | 완료 조건 (Exit Criteria) |
|--------------------------|------------------------|
| 자원 가용성 (인력, 도구, 환경, 데이터, 예산) | 커버리지 수준 달성 |
| 테스트웨어 가용성 (테스트 베이시스, TC) | 미해결 결함 수 |
| 테스트 대상의 초기 품질 (스모크 테스트 합격) | 실패한 TC 수 |
| 예산 승인 | **예산 소진** |
| 테스터 자격 요건 충족 | 리그레션 테스트 자동화 완료 |
| 테스트 환경 준비 | 결함 밀도 도달 |

- 애자일: 완료 조건 = **완료의 정의(DoD)**, 시작 조건 = **준비의 정의(DoR)**

#### 5.1.4 추정 기법 ★★★ (K3)

| 기법 | 유형 | 핵심 |
|------|------|------|
| **비율 기반** | 메트릭 기반 | 과거 프로젝트의 개발:테스트 비율 사용 |
| **외삽법** | 메트릭 기반 | 현재 프로젝트 초기 데이터로 남은 작업 추정. 반복적 SDLC에 적합 |
| **와이드밴드 델파이** | 전문가 기반 | 독립 추정 → 논의 → 재추정 반복. 플래닝 포커는 변형 |
| **3점 추정** | 전문가 기반 | **E = (a + 4m + b) / 6**, SD = (b-a)/6 |

**3점 추정 풀이:** a=낙관, m=유력, b=비관 → 공식 대입 → 단위 작업 추정 × 작업 수

#### 5.1.5 테스트 케이스 우선순위지정 ★★★ (K3)
- **리스크 기반:** 가장 중요한 리스크 TC 먼저
- **커버리지 기반:** 가장 높은 커버리지 TC 먼저 (추가 커버리지 우선순위 변형 있음)
- **요구사항 기반:** 가장 중요한 요구사항 TC 먼저
- **종속성 고려:** 우선순위 높은 TC가 낮은 TC에 종속되면 낮은 것 먼저 실행

**풀이 팁:**
1. 종속성 먼저 해결 (선행 TC 먼저)
2. 그 다음 우선순위 순서로 배치
3. 커버리지 기반이면: 가장 많이 커버하는 TC 선택 → 남은 것 중 추가 커버리지 가장 큰 TC 선택 반복

#### 5.1.6 테스트 피라미드
- 아래층: 작고 빠른 테스트 多 (단위), 위층: 크고 느린 테스트 少 (E2E)
- 테스트 **세분화 수준(granularity)**이 다를 수 있음을 보여줌
- 팀의 **테스트 자동화와 노력 할당** 지원

#### 5.1.7 테스팅 사분면 ★★
| | 팀 지원 | 제품 평가 |
|------|--------|---------|
| **기술** | Q1: 단위, 컴포넌트 통합 (자동화) | Q4: 성능, 보안 등 비기능 (자동화) |
| **비즈니스** | Q2: 기능, 사용자 스토리, API (수동/자동) | Q3: 탐색적, 사용성, UAT (주로 수동) |

### 5.2 리스크 관리 ★★★

#### 5.2.1 리스크 정의
- **리스크 수준 = 발생 가능성 × 영향도**
- 발생 가능성과 영향도는 **독립적** (영향 높다고 가능성도 높은 건 아님)

#### 5.2.2 프로젝트 리스크 vs 제품 리스크
| 프로젝트 리스크 | 제품 리스크 |
|--------------|-----------|
| 조직 문제 (인도 지연, 예산 축소) | 누락/잘못된 기능 |
| 인력 문제 (기술 부족, 갈등) | 부정확한 계산 |
| 기술적 문제 (**개발 범위 증가**, 도구 부족) | 느린 응답시간 |
| 공급업체 문제 | **열악한 아키텍처**, 보안 취약점 |

#### 5.2.3 제품 리스크 분석
- 리스크 식별 + 리스크 평가
- 분석 결과 활용: 테스트 범위/레벨/유형/기법/커버리지 결정, 노력 추정, 우선순위지정

#### 5.2.4 제품 리스크 제어
- **리스크 완화:** 테스팅 수행 (적합한 테스터, 독립성, 리뷰, 적절한 기법/커버리지)
- **리스크 수용:** 통제 불가한 리스크 수용
- **리스크 전가:** 보험 등으로 제3자에게 이전
- **대안 계획:** 리스크 발생 시 대응 계획

### 5.3 테스트 모니터링/제어/완료 ★★

#### 5.3.1 테스팅 메트릭
- 프로젝트 진행: 작업 완료율, 자원 사용률
- 테스트 진행: TC 구현/실행 진행률, 합격/불합격 수
- **제품 품질: 가용성, 응답 시간, 평균 장애 시간(MTTF)**
- 결함: 발견/수정 수, **결함 밀도**, 결함 발견 비율
- 커버리지: 요구사항 커버리지, 코드 커버리지

#### 5.3.2 테스트 보고서
- **테스트 진행 상황 보고서:** 기간, 진행 상황, 방해 요소, 메트릭, 신규 리스크, 다음 계획
- **테스트 완료 보고서:** 요약, 품질 평가, 편차, 방해 요소, 메트릭, 미완화 리스크, 교훈
- 결함 보고서 ≠ 테스트 보고서 (결함 상세 정보는 **결함 보고서**의 목적)

#### 5.3.3 테스팅 상황 전달
- 방법: 대화, 대시보드, 이메일, 온라인 문서, 공식 보고서
- 분산 팀 → 더 공식적 방법 적합
- **번다운 차트:** 반복주기 중 완료/남은 작업량 시각화

---

### 5.4 형상 관리 ★★
- 테스트웨어를 **형상 항목으로 식별, 제어, 추적**
- 베이스라인 → 공식 변경 제어 프로세스로만 수정
- 이전 베이스라인 복원 → **이전 테스트 결과 재현 가능**
- 모든 형상 항목: 고유 식별자, 버전 관리, 변경 추적, 연관성 식별
- **장애 보고 시 소프트웨어 버전으로 릴리스 파일 재구성 가능한 이유 = 형상 관리**

---

### 5.5 결함 관리 ★★★ (K3)
결함 보고서 필수 항목:
- 고유 식별자, 제목, 날짜, 작성자, 테스트 대상/환경 식별 정보
- 결함의 정황 (TC, 활동, SDLC 단계)
- **재현 절차** + 장애 재현/해결에 필요한 정보
- **기대 결과 vs 실제 결과**
- 심각도, 우선순위, 결함 상태, 참조

**시험 포인트:** 결함 보고서에서 "누락된 가장 중요한 정보"를 묻는 문제
- 개발자가 재현하려면: **테스트 환경 정보**(브라우저, OS 등)와 **구체적 입력 데이터**가 핵심
- 결함 설명: 단순 "실패했다"가 아니라 **패턴 분석** (예: "중복 항목을 무시하는 것 같다")

## 📌 제1장 테스팅의 기초 (8문항)

### 1.1 테스팅이란 무엇인가 ★★★

#### 테스트 목적 (자주 출제!)
- 작업 산출물 평가 / 장애 유발 및 결함 식별
- 커버리지 보장 / 리스크 수준 완화
- **베리피케이션:** 정의된 요구사항 충족 확인, 법적/규제 요구사항 준수 확인
- **밸리데이션:** 사용자/이해관계자의 기대 충족 확인
- 정보에 입각한 결정을 위한 정보 제공 / **품질에 대한 자신감 확보**
- 테스트 대상 완성 여부와 이해관계자 기대 충족 확인

**함정:** "결함 식별 및 **수정**" → 수정은 디버깅이지 테스팅이 아님!

#### 테스팅 vs 디버깅
| 테스팅 | 디버깅 |
|--------|--------|
| 장애 유발(동적) 또는 결함 직접 식별(정적) | 장애 원인(결함) 찾기 → 분석 → **제거** |
| 테스터 수행 | 개발자 수행 |
| 결함 **존재 여부** 판단 | 결함 **제거** |

### 1.2 테스팅이 왜 필요한가 ★★

#### 오류 → 결함 → 장애 → 근본 원인
| 용어 | 설명 |
|------|------|
| **오류(Error)** | 사람의 실수 (피로, 시간 압박 등으로 발생) |
| **결함(Defect)** | 오류로 인해 작업 산출물에 생긴 문제 (코드 버그, 명세 오류) |
| **장애(Failure)** | 결함이 실행되어 나타난 **관찰 가능한 문제** |
| **근본 원인** | 오류를 유발한 근본적 이유 (예: 시간 압박) |

**함정 패턴:** 시나리오에서 각 용어를 정확히 구별하는 문제 빈출

#### 테스팅 vs 품질 보증(QA)
- **테스팅 = 품질 제어(QC)의 주요 활동** (제품 지향, 교정 접근법)
- **QA = 프로세스 중심, 예방 접근법** (좋은 프로세스 → 좋은 제품)
- 테스팅 ≠ QA, 테스팅 ≠ QC (QC에는 정형 기법, 시뮬레이션 등도 포함)

### 1.3 테스팅의 7가지 원리 ★★★★

| # | 원리 | 핵심 키워드 |
|---|------|----------|
| 1 | 결함 존재를 밝히지만, 없음을 증명하지 않음 | 결함 부재 증명 불가 |
| 2 | 완벽한 테스팅은 불가능 | 기법/우선순위/리스크로 노력 집중 |
| 3 | **조기 테스팅**으로 시간과 비용 절약 | 정적+동적 모두 일찍 시작 |
| 4 | **결함은 집중**된다 | 파레토 원리, 소수 컴포넌트에 집중 |
| 5 | **테스트 효과는 줄어든다** | 같은 테스트 반복 → 신규 결함 발견↓ |
| 6 | 테스팅은 **정황에 의존적** | 하나의 접근법은 없음 |
| 7 | **결함 부재는 궤변** | 결함 없어도 사용자 불만족 가능 → 밸리데이션도 필요 |

**시험 포인트:** 시나리오 주고 "어떤 원리?" 묻는 문제
- 리그레션 TC 변경 없이 결함 안 나옴 → **원리 5 (테스트 효과 감소)**
- 모든 결함 수정해도 사용자 불만 → **원리 7 (결함 부재 궤변)** → 대처: 밸리데이션(인수 테스팅)

### 1.4 테스트 활동, 테스트웨어, 테스트 역할 ★★★

#### 테스트 활동과 산출물
| 활동 | 주요 업무 | 산출물 |
|------|---------|--------|
| **계획** | 목적 정의, 접근법 선택 | 테스트 계획, 일정, 리스크 관리 대장 |
| **모니터링/제어** | 진행 점검, 조치 | 진행 보고서, 제어 지침 |
| **분석** | 테스트 베이시스 분석 → **테스트 컨디션** 정의 | 우선순위 지정된 테스트 컨디션 |
| **설계** | 컨디션 → **TC**, 커버리지 항목, 환경 요구사항 | TC, 테스트 차터, 커버리지 항목, 테스트 데이터 요구사항 |
| **구현** | TC → **테스트 절차**, 데이터 생성, 환경 구축 | 테스트 절차, 스크립트, 스위트, **테스트 데이터**, 실행 일정 |
| **실행** | 테스트 런, 결과 비교, 결함 보고 | 테스트 로그, 결함 보고서 |
| **완료** | 재사용 테스트웨어 보관, 교훈 파악 | 완료 보고서, **변경 요청서**, 교훈 |

**자주 나오는 문제:** "이 업무는 어떤 활동?" → 테스트 기법 사용 = 분석+설계

#### 테스팅 역할
| 테스트 관리 역할 | 테스팅 역할 |
|----------------|-----------|
| 계획, 모니터링/제어, 완료 | 분석, 설계, 구현, 실행 |
| 테스트 완료 보고서 작성 | 테스트 베이시스/대상 평가 |
| 일정/자원 관리 | 테스트 환경 요구사항 정의 |

### 1.5 필수 기술 및 모범 사례 ★★

#### 전체 팀 접근법
- 장점: **팀 역학 향상**, 의사소통/협업 강화, 시너지 창출
- 모든 팀원이 품질에 책임
- 테스터: 비즈니스 담당자와 인수 테스트 작성, 개발자와 테스트 전략/자동화 협업

#### 테스팅 독립성
| 독립성 수준 | 예시 |
|-----------|------|
| 없음 | 저자가 직접 테스트 |
| 일정 수준 | 같은 팀 동료 |
| 높음 | 팀 외부, 조직 내부 |
| 매우 높음 | 조직 외부 |

- **장점:** 다른 편향 → 다른 유형의 결함 발견, 가정에 의문 제기
- **단점:** 개발팀과 차단 → 협업/소통 어려움, 적대적 관계, 품질 책임감 약화, 병목 비난

## 📌 제2장 SDLC와 테스팅 (6문항)

### 2.1 SDLC에서의 테스팅 ★★★

#### 좋은 테스팅 프랙티스 (모든 SDLC 적용)
1. **모든 개발 활동에 상응하는 테스트 활동** 존재
2. 각 테스트 레벨마다 **구체적이고 독립적인** 테스트 목적
3. 테스트 분석/설계는 **상응하는 개발 단계에서** 시작 (조기 테스팅)
4. 초안 가용 즉시 **작업 산출물 리뷰** 참여 (시프트 레프트)

#### 테스트 우선 접근법
- **TDD:** 테스트 → 코드 → 리팩토링
- **ATDD:** 인수 조건 → 테스트 → 개발
- **BDD:** Given/When/Then → 자동 실행 가능 테스트

#### 데브옵스와 테스팅
- 이점: 빠른 피드백, 시프트 레프트 촉진, 안정적 테스트 환경, 비기능 가시성↑, 자동 리그레션
- 리스크/어려움: 파이프라인 설정, CI/CD 도구 유지보수, **테스트 자동화 구축 어려움**

#### 시프트 레프트
- SDLC **초기에** 테스트 활동 앞당기기 (후반 테스트를 무시하란 뜻 아님)
- 프랙티스: 명세 리뷰, 코드 전 TC 작성, CI/CD, 동적 테스팅 전 정적 분석, 조기 비기능 테스팅

#### 회고
- 성공적이었던 것 / 부족했던 것 / 향후 개선 방법
- 이점: 효과성/효율성 향상, 테스트웨어 품질↑, 팀 결속/학습↑, 개발-테스팅 협업↑
- 참가자: 테스터, 개발자, 설계자, 제품 소유자 (최종 사용자는 보통 불참)

### 2.2 테스트 레벨과 유형 ★★

#### 5가지 테스트 레벨
| 레벨 | 초점 | 수행자 |
|------|------|--------|
| 컴포넌트(단위) | 개별 컴포넌트 | 개발자 |
| 컴포넌트 통합 | 컴포넌트 간 인터페이스/상호작용 | 개발자 |
| 시스템 | 전체 시스템 동작/기능, E2E, 비기능 | 독립 테스트팀 |
| 시스템 통합 | 다른 시스템/외부 서비스와의 인터페이스 | 테스터 |
| 인수 | **밸리데이션**, 배포 준비 확인 | 실제 사용자 |

#### 테스트 유형
- **기능:** "무엇을" 하는가 → 기능 정확성, 완전성, 적합성
- **비기능:** "얼마나 잘" 하는가 → 성능, 호환성, 사용성, 신뢰성, 보안, 유지보수성, 이동성, 안전성

#### 확인 테스팅 vs 리그레션 테스팅
| 확인 테스팅 | 리그레션 테스팅 |
|-----------|--------------|
| 결함이 **성공적으로 수정**됐는지 확인 | 변경으로 **부정적 영향** 없는지 확인 |
| 변경된 코드 대상 | 변경되지 않은 코드 포함 |
| 수정 후 먼저 수행 | 확인 테스팅 후 수행 |

### 2.3 유지보수 테스팅
- 계기: 수정/개선(릴리스 기반), 환경 업그레이드/마이그레이션, **시스템 단종**
- 단종 시: 데이터 보관 테스팅, 복원/복구 절차 테스팅

## 📌 제3장 정적 테스팅 (4문항)

### 3.1 정적 테스팅 기초 ★★

#### 정적 vs 동적 테스팅
| 정적 테스팅 | 동적 테스팅 |
|-----------|-----------|
| 소프트웨어 **실행 안 함** | 소프트웨어 **실행** |
| 결함을 **직접** 식별 | 장애 유발 → 결함 **간접** 식별 |
| 비실행 산출물에도 적용 가능 | 실행 가능 산출물만 |
| 유지보수성 등 코드 실행 불필요 특성 측정 | 성능 등 코드 실행 필요 특성 측정 |
| 도달 불가능 코드, 미사용 변수 식별 | 런타임 오류, 성능/메모리 문제 식별 |

- **정적 테스팅으로만** 찾을 수 있는 것: 도달 불가 코드, 미사용 변수, 설계 불일치
- **동적 테스팅으로만** 찾을 수 있는 것: 응답 시간 지연, 메모리 과다 사용
- **양쪽 모두** 찾을 수 있는 것: 프로그래밍 결함

#### 정적 테스팅 불가능 대상
- 사람이 해석 어려운 것: **암호화된 코드**
- 도구 분석 불가: 법적 문제 있는 타사 실행 코드

### 3.2 피드백과 리뷰 프로세스

#### 리뷰 프로세스 순서
**계획 → 리뷰 착수 → 개별 리뷰 → 논의 및 분석 → 수정 및 보고**

#### 리뷰 역할
| 역할 | 책임 |
|------|------|
| 관리자 | 리뷰 **대상 결정**, 자원 제공 |
| 저자 | 작업 산출물 작성/수정 |
| **중재자** | 회의 효과적 진행, **안전한 환경 조성** |
| 서기 | 이상 사항 수집, 정보 **기록** |
| 리뷰어 | 리뷰 수행 |
| 리뷰 리더 | 참여자 결정, 일정 협의, **전반적 책임** |

#### 리뷰 유형
| 유형 | 공식성 | 주도자 | 주요 목적 |
|------|--------|--------|---------|
| 비공식 리뷰 | 낮음 | - | 이상 사항 식별 (공식 문서 불요) |
| **워크쓰루** | 중간 | **저자** | 품질 평가, 교육, 합의, 새 아이디어 |
| 기술 리뷰 | 중간 | **중재자** | 기술 합의/결정, 이상 사항 식별 |
| **인스펙션** | 높음 | 중재자 | 이상 사항 **최대한 많이** 식별 + **메트릭 수집** |

- 인스펙션: 저자가 리뷰 리더나 서기 **될 수 없음**

#### 리뷰 성공 요소
- 명확한 목표/측정 가능 완료 조건 (**참가자 평가는 절대 안 됨**)
- 적절한 리뷰 유형 선택 / 작은 단위로 분할
- 충분한 준비 시간 / 관리층 지원
- 조직 문화의 일부 / 적절한 교육 / 퍼실리테이션

## 📌 제6장 테스트 도구 (2문항)

### 6.1 테스팅 지원 도구
| 도구 분류 | 역할 |
|---------|------|
| **테스트 관리** | SDLC, 요구사항, 테스트, 결함, **형상 관리** |
| 정적 테스팅 | 리뷰, 정적 분석 지원 |
| 테스트 설계/구현 | TC, 테스트 데이터, 절차 생성 |
| 테스트 실행/커버리지 | 자동 실행, 커버리지 측정 |
| 비기능 테스팅 | 수동 수행 어려운 비기능 테스트 |
| **데브옵스** | 파이프라인, CI/CD, 작업 흐름 추적 |
| **협업** | 의사소통 촉진 |

- **테스트 실행에 도움:** 데브옵스 도구 + 비기능 테스팅 도구

### 6.2 테스트 자동화의 효과와 리스크

| 효과(이점) | 리스크 |
|----------|--------|
| 반복 수작업 감소 → 시간 절약 | 비현실적 기대 |
| 일관성/재현성 → 인적 오류 방지 | 시간/비용/노력 부정확 추정 |
| 객관적 평가(커버리지 등) | 수동이 적합한 곳에 도구 사용 |
| 테스팅 정보 접근 용이 | 도구에 지나친 의존 |
| 실행 시간 단축 | **개발 플랫폼과 호환성 부족** |
| 테스터가 더 효과적 테스트 설계할 시간 확보 | 공급업체 종속성/오픈소스 지원 중단 |
